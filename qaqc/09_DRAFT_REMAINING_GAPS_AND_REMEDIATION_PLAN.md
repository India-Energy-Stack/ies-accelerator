## 5. Recommended execution sequence

**Rationale for ordering:** this sequence is ordered strictly by hard dependency, not by nominal severity or scheduling convenience. Governing constraints: (a) WP-B is the only package needing zero new decisions, so its production runs first, in the current shared worktree; (b) recording an `OD-E` decision is not itself package closure — every owner-gated package still requires an executor to apply the chosen outcome, pass its own QE gate, and reach a checkpoint commit before its row can read CLOSED (§9 condition 3); and (c) only one Spark order may be dispatched or in progress at any time (q08 §2's role contract), and isolated worktrees do not relax that capacity rule. Two technical packages carry an owner-decision precondition on their *scoping*, not merely their closure — WP-P1.2 cannot be scoped or executed until `OD-E3` is recorded, and WP-P1.3-P2.8/WP-P2.2 cannot be scoped or executed until `OD-E7` is recorded — and this sequence states both dependencies explicitly rather than leaving them implicit in §6's per-package "overlaps"/"informs" language.

1. **Dispatch WP-B (Wave B) first, in the current shared worktree.** It is the only package needing zero new decisions — q08 §6–§8 is a complete, self-contained Spark work order, already authorized end to end. It runs first because it is fully unblocked and closes a live, externally-verified 404.
   - *Kill criterion carried from q08:* if the reproduced failing-link count ≠ 4, or any per-directory `*.json` count differs from 2/26/1/10, halt and report — do not improvise a fix.
2. **Seek WP-2b's owner authorization and Fable scoping concurrently with WP-B; hold WP-2b's production in the shared worktree until WP-B's checkpoint.** The authorization request and any scoping have no file dependency on WP-B and may proceed the moment WP-B dispatches. But WP-2b's production (the single-file JSON edit) does not start in the shared worktree until WP-B has passed QE and is checkpoint-committed: WP-B's own acceptance test requires its diff to resolve to exactly its five writable paths, and a concurrent edit landing in the same shared worktree before that checkpoint would contaminate that count. WP-2b's executor lane is Sonnet 5 — it is not a Spark order and does not contend for the one-Spark-order-at-a-time slot WP-B holds. If genuine production concurrency with WP-B is wanted, WP-2b must run in an explicitly isolated worktree/branch with its own separate QE — never in the shared worktree WP-B is still writing to.
3. **Seek all eight `OD-E` decisions concurrently, as decision-only work, starting now.** `OD-E1` through `OD-E8` may all be framed, routed, and deliberated in parallel — none of their substance depends on WP-B, WP-2b, or on any other `OD-E` decision. Recording each decision into `qaqc/11_OWNER_DECISION_LEDGER.md` is not part of that concurrent track: in the current shared worktree, creating the ledger and writing any `OD-E` entry into it wait for WP-B's own QE gate to pass and its checkpoint commit to land, and once unblocked, ledger writes serialize — one writer and one checkpoint at a time — because all eight decisions share this single file. An isolated worktree/branch may prepare a decision record ahead of or alongside that queue, but integrating any prepared record into the ledger is still serial, one entry at a time, each passing its own separate QE gate before the next integrates.
