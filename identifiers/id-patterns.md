# ID Patterns

This page is a cookbook. For each kind of subject in IES — DISCOM, regulator, consumer, asset, meter, credential, dataset — it shows the **exact identifier grammar**, **how it relates to your existing internal numbering**, and **a worked example**.

The rule that ties everything together: a DISCOM's **internal numbering is preserved as the tail of the DID**. You do not throw away your CIS consumer numbers, your SAP asset codes, your meter SLNOs. You wrap them.

---

## Conventions used in this section

| Symbol | Meaning |
|---|---|
| `<discom-id>` | The DISCOM's stable short code in the IES network — `tpddl`, `bescom`, `kseb`, etc. Assigned by the IES network operator when the DISCOM joins the reference registry. |
| `<namespace>` | A DeDi namespace owned by the DISCOM. Most DISCOMs use their `<discom-id>` as the namespace name; large DISCOMs may run sub-namespaces (`tpddl-north`, `tpddl-genco`). |
| `<internal-id>` | The DISCOM's own existing identifier — consumer number, meter SLNO, GIS asset code. Used **verbatim** as the last path segment of a DID. |
| URL-safe | Replace any characters in `<internal-id>` that are not `[A-Za-z0-9._-]` with `%xx` percent-encoding when constructing a DID. Keep a canonical un-encoded form for display only. |

## 1. DISCOM (Issuer)

### Identifier

```
Operational DID :  did:web:<discom-domain>
Trust entry     :  did:dedi:india-energy-stack:ies-discoms-reference-registry:<discom-id>
```

### Example — TPDDL

```
did:web:ies.tpddl.in
did:dedi:india-energy-stack:ies-discoms-reference-registry:tpddl
```

### How it relates to your internal world

| Internal artefact | Maps to |
|---|---|
| Your domain (`tpddl.in`) | Subdomain you publish `.well-known/did.json` on |
| Your company short name (`tpddl`) | `<discom-id>` in the IES reference registry |
| Your CCA-issued DSC | (Optional) An extra key in the DID Document or `x5c` chain in the registry entry |

### Where it goes

```json
"issuer": {
  "id":   "did:web:ies.tpddl.in",
  "name": "Tata Power Delhi Distribution Limited",
  "idRef": {
    "issuedBy":  "did:web:did.cord.network:76EU9AJNL25X4LAxgb92rA8op4co7n892oeySAuEk9gAay2N28ctma",
    "subjectId": "india-energy-stack:tpddl"
  }
}
```

## 2. Regulator

### Identifier

```
Operational DID :  did:web:<regulator-domain>
Trust entry     :  did:dedi:india-energy-stack:ies-regulators-reference-registry:<regulator-id>
```

### Example — Delhi Electricity Regulatory Commission

```
did:web:ies.derc.gov.in
did:dedi:india-energy-stack:ies-regulators-reference-registry:derc
```

Regulators sign tariff orders, ARR approvals, and licensing credentials. The identifier grammar is identical to a DISCOM — only the registry name differs (`ies-regulators-reference-registry` instead of `ies-discoms-reference-registry`).

## 3. Consumer

This is where IES IDs and DISCOM internal IDs meet most directly. The pattern preserves the DISCOM's CIS consumer number end-to-end.

### Two distinct identifiers for the same human

A consumer has **two** identifiers in IES, and they serve different purposes:

| Identifier | Purpose | Generated by | Lifetime |
|---|---|---|---|
| **Holder DID** (`did:key:…`) | Cryptographic identity of the wallet; binds the credential to a key the consumer controls | Wallet / DigiLocker | Lifetime of the key (can rotate by re-issuing credentials to a new key) |
| **Consumer reference ID** (`did:dedi:…`) | Stable DISCOM-side reference for this consumer; what billing systems, asset registries, and data exchange messages route through | DISCOM | Lifetime of the consumer relationship |

The credential carries the holder DID as `credentialSubject.id`, and the consumer reference ID as a property in `credentialSubject.customerProfile.customerReference` (or wherever your schema places it).

### Consumer reference ID — the grammar

```
did:dedi:<discom-namespace>:consumers:<consumer-number>
```

`<consumer-number>` is your existing CIS consumer number, verbatim, URL-safe-encoded.

### Example — TPDDL consumer

Internal CIS consumer number: `TPDDL-2025-001234567`

```
did:dedi:tpddl:consumers:TPDDL-2025-001234567
```

This identifier resolves (via DeDi) to a **consumer reference record** holding:

- The consumer number (canonical form).
- The DISCOM's internal hash of the consumer's contact details (for selective-disclosure-friendly verification — do **not** publish PII).
- Connection metadata that the DISCOM is willing to make resolvable (city, tariff category — never address, phone, name).
- Pointers to currently-active credentials issued to this consumer (hashes, not the credentials themselves).

Whether this record is public on `api.dedi.global` or kept on a **private DeDi mirror** is a DISCOM policy choice — see [Private Resolution](../registries/private_resolution.md).

### Holder DID — generated by the wallet

```
did:key:z6MkjVQ8r4f3rPuY7CG2D6Lf8WJxJBs5sjkR8d3v2Bv4nP4Z
```

You receive this in the credential issuance request — you never generate it. It carries no PII.

### Linking them

In the credential, both appear:

```json
"credentialSubject": {
  "id": "did:key:z6MkjVQ8r4f3rPuY7CG2D6Lf8WJxJBs5sjkR8d3v2Bv4nP4Z",
  "customerProfile": {
    "customerReference": "did:dedi:tpddl:consumers:TPDDL-2025-001234567",
    "customerNumber":    "TPDDL-2025-001234567"
  }
}
```

`customerNumber` stays as the human-readable form (used by call centres, billing letters, DigiLocker UDF1). The `customerReference` DID is what cross-DISCOM workflows route on.

### Why this preserves DISCOM autonomy

- The DISCOM does **not** renumber consumers.
- The DISCOM does **not** allocate new sequences from a national authority.
- A consumer who moves DISCOMs gets a new `customerReference` from the new DISCOM (their old reference is closed in TPDDL's registry); their holder DID can remain the same key.

## 4. Asset (Transformer, Feeder, Substation, Solar Plant)

### Identifier

```
did:dedi:<discom-namespace>:assets:<asset-class>:<internal-asset-id>
```

`<asset-class>` is one of: `feeder`, `transformer`, `substation`, `solar-plant`, `wind-farm`, `bess` (battery storage), `ev-charger`, `meter`. Use kebab-case.

### Example — a TPDDL distribution transformer

Internal asset code in SAP: `DT-NDL-11KV-04572`

```
did:dedi:tpddl:assets:transformer:DT-NDL-11KV-04572
```

### Example — a 220kV substation

```
did:dedi:tpddl:assets:substation:SS-22OK-NJP-001
```

### The asset record

The record this DID resolves to carries:

```json
{
  "id": "did:dedi:tpddl:assets:transformer:DT-NDL-11KV-04572",
  "type": "Transformer",
  "owner": "did:dedi:india-energy-stack:ies-discoms-reference-registry:tpddl",
  "class": "transformer",
  "subclass": "distribution",
  "ratedCapacityKVA": 200,
  "geo": { "lat": 28.6824, "lon": 77.1547 },
  "commissionedAt": "2019-06-12",
  "decommissionedAt": null,
  "endpoints": [{
    "type": "telemetry",
    "becknBPP": "did:web:ies.tpddl.in#beckn-bpp"
  }]
}
```

Geo and commissioning data are typically public; ratepayer-sensitive details (capex, vendor) are not.

### Meters as a special case

A meter is an asset, but it is intimately tied to a service connection. Two valid patterns:

```
# Meter as a standalone asset
did:dedi:tpddl:assets:meter:METER-SLNO-INF7821X9

# Meter as a sub-record of a consumer's connection
did:dedi:tpddl:connections:CONN-TPDDL-2025-001234567:meter
```

Use the standalone form when the meter is the data-exchange subject (telemetry per OpenADR). Use the connection-scoped form when the meter is just a property of a service connection.

## 5. Service Connection

The service connection is the "thing" that gets electricity. It binds a consumer, a meter, and a feeder.

```
did:dedi:<discom-namespace>:connections:<connection-id>
```

`<connection-id>` is whatever your CIS uses — often the same as the consumer number, sometimes a separate SC number.

```
did:dedi:tpddl:connections:CONN-TPDDL-2025-001234567
```

Record links to consumer, meter, feeder, tariff category:

```json
{
  "id": "did:dedi:tpddl:connections:CONN-TPDDL-2025-001234567",
  "consumer": "did:dedi:tpddl:consumers:TPDDL-2025-001234567",
  "meter":    "did:dedi:tpddl:assets:meter:METER-SLNO-INF7821X9",
  "feeder":   "did:dedi:tpddl:assets:feeder:FDR-11KV-NDL-072",
  "tariff":   "DS-1A",
  "status":   "active"
}
```

This is the join table of the IES address space — most cross-domain queries route through here.

## 6. Credential identifiers and revocation handles

A credential has two IDs:

| ID | Where it lives |
|---|---|
| `credential.id` — typically a `urn:uuid:` | Inside the credential JSON, the canonical handle |
| Revocation handle — `did:dedi:<namespace>:vc-revocation-registry:<credential-id>` | In `credentialStatus.id`; what verifiers query to check revocation |

Example:

```json
"id": "urn:uuid:e5f6a7b8-9012-34ab-cdef-567890123456",
"credentialStatus": {
  "id": "https://dedi.global/dedi/lookup/tpddl/vc-revocation-registry/e5f6a7b8-9012-34ab-cdef-567890123456",
  "type": "dedi",
  "statusPurpose": "revocation",
  "statusListCredential": "https://dedi.global/dedi/query/tpddl/vc-revocation-registry"
}
```

The revocation registry is **per-DISCOM**, under the DISCOM's own DeDi namespace. Only the DISCOM's namespace key can write to it.

## 7. Dataset (Beckn `DatasetItem`)

For data-exchange resources (meter telemetry, ARR filings, tariff schedules), use:

```
did:dedi:<discom-namespace>:datasets:<dataset-class>:<dataset-id>
```

Examples:

```
did:dedi:tpddl:datasets:meter-telemetry:2026-01
did:dedi:derc:datasets:tariff-orders:2025-26-domestic
```

The DID resolves to a `DatasetItem` record (DDM schema) carrying the Beckn BPP endpoint that serves the actual data.

## Summary table

| Subject | DID grammar | Internal ID wraps as |
|---|---|---|
| DISCOM | `did:web:<domain>` + `did:dedi:india-energy-stack:ies-discoms-reference-registry:<discom-id>` | Domain + short code |
| Regulator | `did:web:<domain>` + `did:dedi:india-energy-stack:ies-regulators-reference-registry:<regulator-id>` | Domain + short code |
| Consumer | Holder: `did:key:…`; Reference: `did:dedi:<discom>:consumers:<consumer-no>` | CIS consumer number |
| Asset | `did:dedi:<discom>:assets:<class>:<internal-asset-id>` | SAP / GIS asset code |
| Meter | `did:dedi:<discom>:assets:meter:<meter-slno>` | Meter SLNO |
| Connection | `did:dedi:<discom>:connections:<connection-id>` | SC number / consumer number |
| Credential revocation | `did:dedi:<discom>:vc-revocation-registry:<credential-id>` | UUID |
| Dataset | `did:dedi:<discom>:datasets:<class>:<dataset-id>` | Dataset version / range |
