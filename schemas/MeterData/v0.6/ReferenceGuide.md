# MeterData v0.6 Reference Guide

This reference guide details the profiles, canonical register mappings, telemetry modes, and data exchange structures supported by version 0.6 of the India Energy Stack (IES) `MeterData` standard.

---

## 1. Profiles & Core Definitions

The standard organizes data exchanges into eight distinct profile shapes. Each profile serves a specific architectural role:

### A. Customer Profile (`CUSTOMER`)
* **Role**: Captures slow-changing customer metadata, physical service delivery points, and physical meter details.
* **Semantic Constraints**: Omits periodic telemetry values. Connects physical meter installations with utility business entities.
* **Link to Example**: [`CustomerProfile.json`](examples/CustomerProfile.json)

### B. Interval Profile (`INTERVAL`)
* **Role**: Represents block load survey profiles collected at high resolution (typically 15-minute or 30-minute intervals).
* **Semantic Constraints**: Primarily uses `USAGE` mode. Supported registers include block incremental energy registers.
* **Link to Example**: [`IntervalProfile.json`](examples/IntervalProfile.json) (Compact) and [`IntervalProfile_Elaborated.json`](examples/IntervalProfile_Elaborated.json) (Elaborated)

### C. Daily Profile (`DAILY`)
* **Role**: Represents daily accumulated load survey snapshots collected at midnight boundary.
* **Semantic Constraints**: Primarily uses `READING` mode for cumulative registers. 
* **Link to Example**: [`DailyProfile.json`](examples/DailyProfile.json) (Compact) and [`DailyProfile_Elaborated.json`](examples/DailyProfile_Elaborated.json) (Elaborated)

### D. Monthly Profile (`MONTHLY`)
* **Role**: Captures historical billing determinants recorded at the time of a billing reset (typically once a month).
* **Semantic Constraints**: Requires cumulative energy registers (reported in `USAGE` mode with `openingValue` and `closingValue` to prove billing calculations) and Maximum Demand peak summation values.
* **Link to Example**: [`MonthlyProfile.json`](examples/MonthlyProfile.json), [`MDM_MonthlyProfile.json`](examples/MDM_MonthlyProfile.json) (Multi-meter), and [`MonthlyProfile_MultipleResets.json`](examples/MonthlyProfile_MultipleResets.json) (Ad-hoc resets)

### E. Bill Details (`BILL_DETAILS`)
* **Role**: Represents commercial utility billing computed details generated by Customer Information Systems (CIS) or billing engines.
* **Semantic Constraints**: Contains monetary figures, bill dates, due dates, prepaid balances, and payment statuses. It may reference the corresponding MonthlyProfile data.
* **Link to Example**: [`BillDetails.json`](examples/BillDetails.json), [`Billing_MonthlyProfile.json`](examples/Billing_MonthlyProfile.json), and [`Billing_MeterChange.json`](examples/Billing_MeterChange.json)

### F. Instantaneous Profile (`INSTANTANEOUS`)
* **Role**: Captures a point-in-time snapshot of electrical quantities.
* **Semantic Constraints**: Typically triggered ad-hoc. Includes snapping voltage, current, active/reactive power, and grid frequency.
* **Link to Example**: [`InstantaneousProfile.json`](examples/InstantaneousProfile.json)

### G. Event Profile (`EVENT`)
* **Role**: Represents logged event alerts matching diagnostic and tamper thresholds.
* **Semantic Constraints**: Emits precise timestamps and standard IS 15959 event category IDs.
* **Link to Example**: [`EventProfile.json`](examples/EventProfile.json)

### H. Alarm Profile (`ALARM`)
* **Role**: Represents active alarms currently triggered on the meter.
* **Semantic Constraints**: Exchanged for near real-time state flags (such as active power failure, low prepaid balance, or cover open tamper).
* **Link to Example**: [`AlarmProfile.json`](examples/AlarmProfile.json)

## 2. Centralised Code List & Payload Descriptors

To decouple profile payloads from redundant metadata overhead, the standard uses a centralized registry mapped through local descriptors.

### Centralized Code List (`IES codes.json`)
The registry file [`IES codes.json`](IES%20codes.json) serves as the canonical catalog for all registers. It maps OBIS codes (e.g. `1.0.1.8.0.255`) and Short Names (e.g. `kWh imp`) to:
- **`unit`**: The standard unit of measure (e.g. `kWh`, `V`, `kW`).
- **`flowDirection`**: `IMPORT`, `EXPORT`, or `NONE`.
- **`accumulationBehaviour`**: `CUMULATIVE`, `DELTA`, `SUMMATION`, or `INDICATING`.
- **`supportedModes`**: `["READING", "USAGE"]`.
- **`defaultMode`**: The default telemetry mode (e.g., `READING` for energy accumulators).

### Payload Descriptors & Metadata Resolution
When using the compact matrix array format (Form B), profiles define local metadata sets via `payloadDescriptors`:
- **`readingType`**: The short code or OBIS code naming the register.
- **`reportedMode`**: Must be declared inside the descriptor (either `READING` or `USAGE`). This mode dictates whether attributes like `openingValue`/`closingValue` are permitted.
- **`multiplier`**: Decimal scaling factor (e.g. `0.001` for milli, `1000` for kilo) applied to values. The default value is `1`.
- **`accuracy`**: The accuracy class or precision value, applied after the multiplier.

## 3. Telemetry Data Specifications

### Cumulative Energy Registers
- **Accumulation Behaviour**: `CUMULATIVE`
- **Telemetry Modes**: `READING` (raw dial reading snapshot) or `USAGE` (consumption over interval/billing reset, requiring `openingValue` and `closingValue` verification attributes).
- **Core Registers**:
  - `1.0.1.8.0.255` (`kWh imp`): Active energy import.
  - `1.0.2.8.0.255` (`kWh exp`): Active energy export.
  - `1.0.9.8.0.255` (`kVAh imp`): Apparent energy import.

### Block Incremental Energy Registers
- **Accumulation Behaviour**: `DELTA`
- **Telemetry Modes**: Strictly `USAGE`.
- **Core Registers**:
  - `1.0.1.29.0.255` (`kWh imp block`): Interval active import.
  - `1.0.9.29.0.255` (`kVAh imp block`): Interval apparent import.

### Maximum Demand Registers
- **Accumulation Behaviour**: `SUMMATION` / `INDICATING`
- **Telemetry Modes**: Strictly `USAGE`.
- **Core Registers**:
  - `1.0.1.6.0.255` (`MD kW`): Import active power demand peak over integration window.
  - `1.0.9.6.0.255` (`MD kVA`): Import apparent power demand peak.
- **Attributes**: MD records require peak occurrence time (`occurredAt`) and demand integration duration (`integrationPeriod`, e.g. `PT30M`).

## 4. Non-Telemetry Data Specifications

Non-telemetry data fields are reserved for commercial, identity, and billing variables:
- **`customerRefs`**: Business identifiers (e.g. `CONSUMER_NUMBER` or Account ID).
- **`meterRefs`**: Physical meter identifiers (e.g. `METER_SERIAL`).
- **`serviceDeliveryPointRefs`**: Physical grid connection point identifiers (`SERVICE_DELIVERY_POINT`).
- **`amountDue` / `prepaidBalance`**: CIS calculated financial states in the specified currency.

---

## 5. Handling of PII Data

To protect customer privacy and comply with data security requirements, payloads are decoupled and restricted:

1. **Decoupled Telemetry (HES / MDM)**: Payloads originating from HES and MDM networks (such as [`IntervalProfile.json`](examples/IntervalProfile.json) and [`DailyProfile.json`](examples/DailyProfile.json)) intentionally omit `customerRefs`. They only identify data via physical references (`meterRefs` or `serviceDeliveryPointRefs`).
2. **PII Isolation**: Commercial identifiers (`customerRefs`) reside exclusively within billing-related envelopes (such as [`Billing_MonthlyProfile.json`](examples/Billing_MonthlyProfile.json) or [`BillDetails.json`](examples/BillDetails.json)) generated within secure CIS networks.
3. **Anonymization and Hashing**: Systems are encouraged to hash or use transient pseudonyms for `value` properties within identifier arrays (e.g. `customerRefs`) when exchanging data with third-party service providers.
